Chess move generation

Project to perform move generation in Chess to chosen depth. Initial task to complete in order to build a chess bot.

Link to repo

Januuary 2022 10 min read

Purpose of move generation

Move generation is performed in chess bots and engines to determine what moves are available. These moves are then evaluated by the bot or engine to determine the best move to make. In order for chess bots and engines to make a move quickly, move generation must be done quickly as evaluation requires certain amount of time as well. As such, I was trying out various methods to perform move generation is chess to a certain depth. This meant trying out various algorithms to perform move generation accurately. I also compared the optimised versions of the algorithms based on the time required to generate moves from various positions.

Defining the task

Depth in move generation refers to how many moves ahead by both players we are looking at. In the starting position, move generation at a depth of 1 means we consider all possible moves that White can make, which is 20. A depth of 2 means we consider all of White's moves, and all possible responses by Black. We then return the sum of possible responses by Black to each of White's possible moves, which is 400. This continues until we reach the desired depth.

Naive approach

For the first version of the algorithm, I used a simple approach that combined the logic and OOP structure of the chess game written for Pygame. Each piece was represented by a class and the chess board state was stored in an array.The game already had a move generation logic, albeit at a depth of one for the current player. There was work to be done to update the logic to generate moves recursively until a certain depth. As mentioned before, I could reference Perft results on Chess Programming Wiki to test my move generation on various tricky positions.

One of the tricky positions from the wiki

In this position, white is unable to advance the pawn on b5 as it would lead to a discovered check due to the black rook on h5. In addition, there would be double checks at higher depth, which limits the number of valid moves as the king needs to avoid being attacked by multiple pieces in the following move. Handling all these positions took a bit of debugging in order to correct my algorithm. It was extremely helpful to check the results of my algorithm against Stockfish. It can return the correct result in less than a millisecond, which was demoralising as I was really far off compared to its performance.

Performance

As the first version was more of getting it correct than being able to run quickly, it was no surprise that the time taken for move generation was REALLY slow. I benchmarked my algorithm on my PC with Ryzen 5 3600 CPU, 16 GBs of RAM and SSD. Generating moves to a depth of 3 from the initial position took about 4.5 seconds, which was downright unacceptable. I used Pycharm's profiler to look at the main operations of the algorithm, and realised that deepcopying the board array took a significant amount of time. As such, the next version of the algorithm will try to reduce deepcopy operations.

Version 2

For the next version, I used a string to represent the board's FEN instead of an array, as a string is immutable and can be assigned to another variable by duplication. This change alone significantly reduced the amount of deepcopy operations and sped up the algorithm by a factor of 10. I also implemented more helper functions to check if a piece can capture another and if the king is in check.

Version 3

For the next version, I decided to precompute the moves for knight and king pieces for each of the 64 squares. Having a lookup table for these moves meant much fewer calculations were performed during move generation, at the expense of using more space. Instead of using a single string to represent the board, I now used separate strings to store white and black piece positions, as well as king positions. Doing so allowed me to perform operations directly on the strings, instead of having to decode the FEN string repeatedly at every step. These changes brought another tenfold improvement in algorithm runtime for move generation.

Comparison

Most chess games usually ends when a player is checkmated. This occurs when a player's king is in check and has no moves to get out of check. When the game is over, the subsequent turn of the game is disabled and the players can no longer make any moves. The other way a game typically end is stalemate, where a player has no legal moves but is not in check, which results in a draw. The possibility of stalemate meant that I needed to check what legal moves a player has at the start of every turn. This needs be done regardless as the player can click on each piece during his turn to see all possible moves he can make.

Starting position of a chess game

Gameplay

In terms of gameplay, the player with the white pieces start first from the board configured to the default starting position. Since both players cannot skip their turn, they take turns making one move each until the game reaches a decisive result. At the start of a player's turn, all possible moves are stored in a dictionary, where the key is an integer representing the position of a piece relative to the grid, and the values are also integers representing possible squares that piece can move to. The integers 0 to 63 is just an alternative way to represent the squares of a chessboard, where the integer's value is equal to 8 x rank + file. The game also checks if a player is in check, can castle either side or if en passant is playable. When the user clicks on a piece, the coordinate of the mouse relative to the game window is then translated into a square on the chessboard. If the selected piece has possible moves, the game window refreshes to show indicators on the square that the chosen piece can move to.

Screen grab of game

Bishop on a5 selected and possible moves are shown

Assuming the user clicks on a valid square that the selected piece can move to, the game then refreshes to show the updated board and allows the other player to then make a move. If the user changes his mind about which piece he wishes to move, the game window then refreshes to show the valid moves of another selected piece. After every turn, the game updates the availability of castling and en passant.

Final thoughts

The chess game works as intended with the exception of handling certain rare scenarios, such as the 50-move rule and threefold repetition. Future improvements could be made to implement these rules, as well as adding a timer for both players. In addition, the user could be allowed to configure his preferred board and piece theme, along with the animation speed of piece movement and other UI settings. (This would basically be making a clone of Lichess)

Conclusion

Creating a working chess game concludes the first part of making a chess bot. In all, it took about 500 lines of code and way too much time to programme the chess pieces, board and gameplay. The next part of the project would be working on move generation for multiple moves ahead in order to evaluate them. An overview of this is covered in the Move Generation article.
